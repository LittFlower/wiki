## AIxCC 第三名：Theori / RoboDuck 的公开方案

Theori 设计的这套 CRS 叫做 RoboDuck，他们自己把它描述成一个偏 LLM-first 的系统：尽量让 LLM 参与漏洞发现、PoV 生成、补丁生成、crash 根因分析等环节，同时保留 fuzzing 和静态分析作为信号来源与兜底。

### 1. 发现漏洞

RoboDuck 的第一层是**候选发现**。Theori 用了 Infer 这类静态分析工具，也用了 libFuzzer harness 和 coverage 反馈；但传统静态分析的缺陷是在这种场景下误报率很高，因此不能直接把所有报告都当成漏洞。

### 2. 成本过滤

使用成本比较低的 LLM 对第一步发现的漏洞做评分筛选，再把排名靠前的一部分交给更贵的 agent 深度分析。

### 3. PoV 生成

它不是让一个 agent 从头到尾直接写触发样例，而是拆成几个子任务：

1. 输入格式理解
2. 输入编码器生成
3. 源码问题回答
4. 调试失败 Pov
5. 多模型并行尝试
6. 真实运行验证

让 agent 写一个类似 “input encoder” 的 Python 函数，把结构化参数编码成目标程序需要的二进制输入；Pov producer 可以调用 source-question agent、debug agent、test runner 等工具。多个模型会并行尝试，某个 Pov 被真实运行验证成功后，其它尝试可以终止。

LLM 在“语义推理”和“写小工具”上强，但在长链路无反馈推理上不稳定；所以要给它短循环反馈。

### 4. 用 LLM 解决 fuzzing 卡分支问题

先定位 fuzz blocker，再让 LLM 理解输入格式，生成能走到目标分支的种子。

重点是要用 coverage feedback 来降低模型幻觉，通过这种回归验证提高 LLM 解决 fuzzing block 的效率

### 5. 生成补丁

RoboDuck 的 patch agent 会生成 diff，然后跑验证：既要确认原 PoV 被修掉，也要尽量保证原有测试不回归。

**总结**：基本上就是把静态分析、动态分析找🕳这个环节拆得更细一点，并且及时给 agent 反馈，多 agent 协同合作。

## Xint 白皮书

《You Don’t Need Mythos. You Need a System.》

Xint Code 的 pipeline 大致有四层：

1. Attack surface identification / targeting：自动识别攻击面和高价值代码路径，
2. Deep reasoning over candidate code paths：对候选路径做数据流、控制流、跨函数语义分析
3. Exploitability validation：验证问题是否真的可触发、是否有安全影响
4. Structured results：输出 CVSS、复现步骤、影响说明、工作流集成信息
