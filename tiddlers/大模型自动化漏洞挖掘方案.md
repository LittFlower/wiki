## AIxCC 第三名：Theori / RoboDuck 的公开方案

Theori 设计的这套 CRS 叫做 RoboDuck，他们自己把它描述成一个偏 LLM-first 的系统：尽量让 LLM 参与漏洞发现、PoV 生成、补丁生成、crash 根因分析等环节，同时保留 fuzzing 和静态分析作为信号来源与兜底。

### 1. 发现漏洞

RoboDuck 的第一层是**候选发现**。Theori 用了 Infer 这类静态分析工具，也用了 libFuzzer harness 和 coverage 反馈；但传统静态分析的缺陷是在这种场景下误报率很高，因此不能直接把所有报告都当成漏洞。

他们还做了一些 LLM 分析：主要有传统静态分析、LLM 静态分析、fuzzing 这三类

1. 传统静态分析一般就是指比较常见的这几类静态分析工具，[[SARIF]] 复核是让 LLM 复核传统静态分析工具的输出。
2. LLM 静态分析分为单函数分析、大代码块分析、diff 分析
	- 单函数分析：顾名思义，就是一次只喂给 LLM 一个函数，不考虑上下文，只让 LLM 分析当前这个函数是否存在漏洞
	- 大代码块分析：先用程序分析工具选上下文，一个比较好的上下文组织方式如下

	```
	[任务说明]
	请判断下面代码路径是否存在可由外部输入触发的内存安全/逻辑漏洞。
	
	[入口]
	fuzz_harness.cc::LLVMFuzzerTestOneInput
	
	[调用链]
	LLVMFuzzerTestOneInput → parse_file → parse_chunks → decode_payload
	
	[关键结构体]
	struct Header ...
	struct Chunk ...
	
	[代码片段]
	1. harness
	2. parse_file
	3. parse_header
	4. parse_chunks
	5. decode_payload
	
	[要求输出]
	bug_type / trigger_condition / evidence / confidence / next_step
	```
	
	- diff 分析：用来分析单个 diff 的安全影响，攻击面更小、分析更精准
3. fuzzing：用 LLM 根据 coverage 辅助调整，构造一个基于 LLM 的 agent 使用 coverage 数据寻找大段未到达代码，并尝试为这些代码构造输入；即使已有 corpus 覆盖率饱和，这个 agent 也经常能生成带来新 coverage 的 seed。



### 2. 成本过滤

使用成本比较低的 LLM 对第一步发现的漏洞做评分筛选，再把排名靠前的一部分交给更贵的 agent 深度分析。

### 3. PoV 生成

它不是让一个 agent 从头到尾直接写触发样例，而是拆成几个子任务：

1. 输入格式理解
2. 输入编码器生成
3. 源码问题回答
4. 调试失败 Pov
5. 多模型并行尝试
6. 真实运行验证

让 agent 写一个类似 “input encoder” 的 Python 函数，把结构化参数编码成目标程序需要的二进制输入；Pov producer 可以调用 source-question agent、debug agent、test runner 等工具。多个模型会并行尝试，某个 Pov 被真实运行验证成功后，其它尝试可以终止。

LLM 在“语义推理”和“写小工具”上强，但在长链路无反馈推理上不稳定；所以要给它短循环反馈。

### 4. 用 LLM 解决 fuzzing 卡分支问题

先定位 fuzz blocker，再让 LLM 理解输入格式，生成能走到目标分支的种子。

重点是要用 coverage feedback 来降低模型幻觉，通过这种回归验证提高 LLM 解决 fuzzing block 的效率

### 5. 生成补丁

RoboDuck 的 patch agent 会生成 diff，然后跑验证：既要确认原 PoV 被修掉，也要尽量保证原有测试不回归。

**总结**：基本上就是把静态分析、动态分析找🕳这个环节拆得更细一点，并且及时给 agent 反馈，多 agent 协同合作。

```
                ┌──────────────────┐
                │  项目源码 / diff  │
                └─────────┬────────┘
                          │
        ┌─────────────────┼─────────────────┐
        │                 │                 │
        ↓                 ↓                 ↓
  单函数分析         大代码块分析          diff 分析
        │                 │                 │
        └─────────┬───────┴─────────┬───────┘
                  ↓                 ↓
            候选漏洞池          SARIF 报告池
                  ↓                 ↓
            LLM scorer        SARIF 复核
                  └────────┬────────┘
                           ↓
                    高置信 bug candidates
                           ↓
                    PoV / fuzz 验证
                           ↓
                  coverage 卡点分析辅助
                           ↓
                     触发输入 / crash
                           ↓
                    patch agent 修复
                           ↓
                    regression 验证
```

## Xint 白皮书

《You Don’t Need Mythos. You Need a System.》

Xint Code 的 pipeline 大致有四层：

1. Attack surface identification / targeting：自动识别攻击面和高价值代码路径，
2. Deep reasoning over candidate code paths：对候选路径做数据流、控制流、跨函数语义分析
3. Exploitability validation：验证问题是否真的可触发、是否有安全影响
4. Structured results：输出 CVSS、复现步骤、影响说明、工作流集成信息

## Agent Flow

来自论文《Synthesizing Multi-Agent Harnesses for Vulnerability Discovery》

论文里的这个 AgentFlow 着重优化了 CRS 的 agent 挖🕳部分，优化思路如下：

把多 Agent 系统表示成一个带类型检查的图 DSL：节点是 Agent，边是数据流或 retry link，prompt 模板决定每个 Agent 能看到哪些上游输出和哪些运行时反馈。然后它用一个外层优化循环不断改写这个图：Propose 新 harness，Execute 执行，Score 打分，Diagnose 根据 stderr、coverage、sanitizer report 等反馈分析失败原因，再指导下一轮修改。

因为漏洞触发通常是稀疏反馈问题。单纯 pass/fail 不知道失败原因，但 coverage 可以告诉我们输入有没有到达目标函数，sanitizer 可以告诉我们是不是实际内存错误，stderr 可以告诉我们是不是文件格式都没过。AgentFlow 本质上就是把这些运行时信号用于优化 Agent 团队结构，而不是只让一个 Agent 盲目重试。

具体来说，Agent Flow 有以下五层抽象：

1. Agent set：analyst、explorer、verifier 这些角色；
2. topology 决定谁把结果传给谁；
3. message schema 决定下游 Agent 的 prompt 能引用哪些上游结果和哪些反馈；
4. tool allocation 决定某个 Agent 能不能读源码、编译、执行目标；
5. coordination protocol 则表示串行、并行 fanout、失败 retry 等执行方式。

通过反馈动态修改图 DSL，系统可以搜索“是否可以新增一个 verifier，给它 sanitizer 输出，并在失败时把结果反馈给 analyst”这种问题。

另一个，在反馈修改上，由于漏洞触发本来就是稀疏反馈问题，同样是失败，可能是输入格式没过，也可能是没到目标函数，或者到了函数但没进漏洞分支。AgentFlow 的反馈修改逻辑不是简单的 pass/fail，而是 Propose、Execute、Score、Diagnose：先提出新的 harness，执行后收集反馈，用 pass rate 或 unique crash signature 打分，再由 diagnoser 根据完整反馈判断瓶颈，指导下一轮改 prompt、加 Agent、改拓扑、加 fanout 或 retry。

## 一些设计原则

1. LLM 不直接判断这里有🕳然后报告，这样做误报率/幻觉太高，而是当 hypothesis generator，先产生假设，然后用工具验证。
2. 不要直接给个 bash 让 agent 自由发挥，而是给几个 `read_definition`、`find_references`、`read_source` 这种结构化源码工具，背后接 clang AST、Joern、gtags 等索引
3. 先做 delta-mode，再做 full-scan，delta-mode 就是先审计 PR/commit 带来的攻击面
4. 先用低成本 LLM 筛选候选漏洞，选择高价值潜在漏洞点做深入分析
5. PoV、patch、report 都要 artifact 化

```
candidate_id
root_cause_location
trigger_conditions
input_encoder
reproducer / PoV
crash log
sanitizer report
coverage trace
patch diff
patch validation result
test result
report markdown
disclosure status
```
