# V8 3.16 GC 源码逐步导读

本文基于当前目录 `/home/flower/ctf/problem/v8` 的 V8 `3.16.0` 源码，按 GC 主调用链从入口到细节逐步解释：

```text
Heap::CollectGarbage
  -> Heap::PerformGarbageCollection
    -> Heap::Scavenge
    -> Heap::MarkCompact
         -> MarkCompactCollector::CollectGarbage
              -> MarkLiveObjects
              -> SweepSpaces
              -> EvacuateNewSpaceAndCandidates
```

目标不是一次读完所有 visitor 模板，而是先建立完整主线：一次 GC 如何被触发、如何选择 young/full GC、如何复制或标记对象、如何更新引用、为什么需要 write barrier 和 store buffer。

## 0. 先建立对象和空间模型

V8 3.16 的空间枚举在 `src/v8globals.h:174`：

```cpp
enum AllocationSpace {
  NEW_SPACE,
  OLD_POINTER_SPACE,
  OLD_DATA_SPACE,
  CODE_SPACE,
  MAP_SPACE,
  CELL_SPACE,
  LO_SPACE,
};
```

各空间含义：

- `NEW_SPACE`：新生代，使用两个 semispace，young GC 用复制算法。
- `OLD_POINTER_SPACE`：老生代指针对象，可能包含指向新生代的指针。
- `OLD_DATA_SPACE`：老生代纯数据对象，设计上不应包含指向新生代的普通对象指针。
- `CODE_SPACE`：可执行代码对象。
- `MAP_SPACE`：只放 `Map` 对象，也就是 V8 的隐藏类/对象布局描述。
- `CELL_SPACE`：全局属性 cell 等小对象。
- `LO_SPACE`：large object space，大对象不适合在普通页内移动。

所有堆对象继承自 `HeapObject`，见 `src/objects.h:1356`。每个 `HeapObject` 的第一个 word 是 `map word`：

```cpp
static const int kMapOffset = Object::kHeaderSize;
static const int kHeaderSize = kMapOffset + kPointerSize;
```

正常执行时，`map word` 是 `Map*`。GC 期间它可能被临时复用：

- Scavenge 复制后，from-space 旧对象的 `map word` 会变成 forwarding address。
- Full GC 标记/压缩期间，mark bit 和 forwarding 信息通过 page bitmap、slot buffer 等结构配合维护。

这是读 GC 的第一个关键点：对象首 word 不总是普通 `Map*`，GC 状态会改变它的解释方式。

## 1. `Heap::CollectGarbage`：GC 调度总入口

位置：`src/heap.cc:597`

这个函数是 GC 的统一入口。很多路径最终都会走到这里，例如：

- allocation 失败后请求 GC；
- JS 里通过 `--expose_gc` 调用 `gc()`；
- idle notification 或内存压力触发；
- 内部为了保守推进 heap 状态主动触发。

核心职责：

```cpp
bool Heap::CollectGarbage(AllocationSpace space,
                          GarbageCollector collector,
                          const char* gc_reason,
                          const char* collector_reason)
```

关键点：

1. 进入 `VMState state(isolate_, GC)`，让 VM 知道当前处于 GC 状态。
2. 如果原本要做 full GC，但 incremental marking 还在跑，会先推进一小步：

```cpp
incremental_marking()->Step(kStepSizeWhenDelayedByScavenge,
                            IncrementalMarking::NO_GC_VIA_STACK_GUARD);
```

3. 如果推进后还没完成，就把本次 full GC 降级成 scavenge：

```cpp
collector = SCAVENGER;
collector_reason = "incremental marking delaying mark-sweep";
```

这说明 V8 3.16 已经有“用增量标记推迟 full GC 停顿”的思想。不是所有 full GC 请求都会立刻 stop-the-world 做完整 mark-sweep。

随后它创建 `GCTracer`，执行：

```cpp
GarbageCollectionPrologue();
PerformGarbageCollection(collector, &tracer);
GarbageCollectionEpilogue();
```

`CollectGarbage` 自己不负责具体算法，它负责决策、状态管理、回调、统计和收尾。

## 2. `Heap::PerformGarbageCollection`：选择 Scavenge 还是 MarkCompact

位置：`src/heap.cc:851`

这个函数是真正的分派点：

```cpp
if (collector == MARK_COMPACTOR) {
  MarkCompact(tracer);
  ...
} else {
  tracer_ = tracer;
  Scavenge();
  tracer_ = NULL;
  ...
}
```

两条路径：

- `SCAVENGER`：young GC，只处理新生代和 old-to-new 引用。
- `MARK_COMPACTOR`：full GC，处理老生代、新生代、弱引用、清扫、压缩和引用更新。

这个函数还会更新 survival rate：

```cpp
UpdateSurvivalRateTrend(start_new_space_size);
```

survival rate 会影响新生代大小和晋升策略。如果新生代对象大量存活，V8 会认为 mutator 正在构建长生命周期结构，于是调大老生代限制，减少频繁 full GC。

## 3. Young GC 主线：`Heap::Scavenge`

位置：`src/heap.cc:1254`

`Scavenge()` 是新生代 GC，实现 Cheney copying algorithm。

### 3.1 Scavenge 开始前

开头：

```cpp
RelocationLock relocation_lock(this);
gc_state_ = SCAVENGE;
LOG(isolate_, ResourceEvent("scavenge", "begin"));
```

这里进入 scavenge 状态，并加 relocation lock，避免搬迁期间出现不一致。

之后会做几件准备：

```cpp
CheckNewSpaceExpansionCriteria();
SelectScavengingVisitorsTable();
incremental_marking()->PrepareForScavenge();
AdvanceSweepers(static_cast<int>(new_space_.Size()));
```

含义：

- 检查是否需要扩张 new space。
- 根据是否 incremental marking、是否 logging/profiling，选择不同的 scavenge visitor 表。
- 如果 incremental marking 正在进行，先准备好和 scavenge 的协作状态。
- 推进老生代 lazy sweeping，因为 young GC 可能会晋升对象到老生代，需要可用空间。

### 3.2 Flip semispace

核心操作：

```cpp
new_space_.Flip();
new_space_.ResetAllocationInfo();
```

Scavenge 前，活跃 semispace 是当前 to-space。Flip 后：

- 旧 to-space 变成 from-space，里面有待检查对象。
- 新 to-space 为空，用来接收存活对象。
- from-space 中无法到达的对象不用逐个 free，整个 semispace 后续可整体回收。

### 3.3 用 to-space 当 Cheney 队列

初始化：

```cpp
Address new_space_front = new_space_.ToSpaceStart();
promotion_queue_.Initialize();
```

这里有两个队列：

- to-space 底部 `[new_space_front, new_space_.top())` 是已复制但还没扫描字段的对象队列。
- promotion queue 存放已经晋升到 old space、但字段还没完全扫描的对象。

每复制一个对象到 to-space，`new_space_.top()` 前进；`DoScavenge()` 会从 `new_space_front` 往前扫描，直到追上 `top()`。

### 3.4 从 roots 复制

```cpp
ScavengeVisitor scavenge_visitor(this);
IterateRoots(&scavenge_visitor, VISIT_ALL_IN_SCAVENGE);
```

`ScavengeVisitor` 的逻辑在 `src/heap.cc:1046` 附近：

```cpp
void ScavengePointer(Object** p) {
  Object* object = *p;
  if (!heap_->InNewSpace(object)) return;
  Heap::ScavengeObject(reinterpret_cast<HeapObject**>(p),
                       reinterpret_cast<HeapObject*>(object));
}
```

它只关心指向 new space 的指针。如果 root 指向 old space，不需要 young GC 处理。

如果 root 指向 from-space 对象：

1. 复制或晋升该对象。
2. 在 from-space 旧对象上写 forwarding address。
3. 把 root slot 改成新地址。

### 3.5 从老生代引用复制：StoreBuffer

Young GC 不扫描整个 old space，否则就失去分代 GC 的意义。它依赖 write barrier 记录 old-to-new 指针：

```cpp
store_buffer()->IteratePointersToNewSpace(&ScavengeObject);
```

路径：

- 写字段时，如果 old object 指向 new object，会调用 `Heap::RecordWrite()`。
- `Heap::RecordWrite()` 把 slot 放进 `StoreBuffer`。
- Young GC 只遍历这些 remembered slots。

相关源码：

- `src/objects-inl.h:923`：`WRITE_BARRIER`
- `src/heap-inl.h:353`：`Heap::RecordWrite`
- `src/store-buffer.cc:608`：`StoreBuffer::IteratePointersToNewSpace`

这个机制是分代 GC 的正确性核心：如果没有 store buffer，old object 指向 new object 的引用可能被 young GC 漏掉，导致仍可达的新生代对象被误回收。

### 3.6 `Heap::DoScavenge`：推进 Cheney 队列

位置：`src/heap.cc:1649`

主循环：

```cpp
while (new_space_front != new_space_.top()) {
  HeapObject* object = HeapObject::FromAddress(new_space_front);
  new_space_front += NewSpaceScavenger::IterateBody(object->map(), object);
}
```

解释：

- `new_space_front` 指向下一个待扫描对象。
- `new_space_.top()` 指向 to-space 当前分配末尾。
- 扫描对象字段时，如果发现字段指向 from-space 对象，会继续复制或晋升，可能让 `top()` 继续前进。
- 直到 `front == top`，说明所有已复制对象的字段都扫描完。

随后处理 promotion queue：

```cpp
while (!promotion_queue()->is_empty()) {
  promotion_queue()->remove(&target, &size);
  IterateAndMarkPointersToFromSpace(target->address(),
                                    target->address() + size,
                                    &ScavengeObject);
}
```

晋升对象已经在 old space，不能放进 to-space 队列，所以通过 promotion queue 另行扫描。

### 3.7 ScavengeObject 和 EvacuateObject

`Heap::ScavengeObjectSlow()` 在 `src/heap.cc:2120`：

```cpp
MapWord first_word = object->map_word();
Map* map = first_word.ToMap();
map->GetHeap()->DoScavengeObject(map, p, object);
```

真正复制逻辑在 `ScavengingVisitor`，位置 `src/heap.cc:1728`。

核心函数是 `EvacuateObject`：

```cpp
if (heap->ShouldBePromoted(object->address(), object_size)) {
  // allocate in old space
} else {
  // allocate in new to-space
}
```

晋升条件在 `src/heap-inl.h:333`：

```cpp
return below_mark || (new_space_.Size() + object_size) >=
                    (new_space_.EffectiveCapacity() >> 2);
```

含义：

- 对象已经熬过一次 scavenge，即在 age mark 以下，则晋升。
- 或者 to-space 已经达到有效容量的 1/4，为减少复制压力，开始更积极晋升。

复制后：

```cpp
heap->CopyBlock(target->address(), source->address(), size);
source->set_map_word(MapWord::FromForwardingAddress(target));
```

这就是 forwarding pointer 的来源。以后再遇到同一个 from-space 对象，不需要重复复制，只要读 forwarding address 更新 slot。

### 3.8 Scavenge 收尾

Scavenge 末尾会处理：

- object groups 和 weak independent handles；
- external string table；
- error object list；
- runtime profiler samples；
- incremental marking deque；
- weak references；
- 设置新的 age mark；
- 更新 young survivor 统计。

关键不变量：

- 所有可达新生代对象已经在 to-space 或 old space。
- 所有 root/old-to-new slots 已经改成新地址。
- from-space 可以整体视为垃圾。
- `new_space_.age_mark()` 被设置为当前 top，用于下次判断哪些对象已经存活过一次。

## 4. Full GC 入口：`Heap::MarkCompact`

位置：`src/heap.cc:1007`

这个函数是 full GC 的 Heap 层包装：

```cpp
gc_state_ = MARK_COMPACT;
mark_compact_collector_.Prepare(tracer);
MarkCompactPrologue();
mark_compact_collector_.CollectGarbage();
gc_state_ = NOT_IN_GC;
```

职责分工：

- `Prepare()`：初始化 MarkCompactCollector，选择 evacuation candidates。
- `MarkCompactPrologue()`：清理 keyed lookup cache、context slot cache、descriptor lookup cache、regexp cache、number string cache 等，避免缓存强保活无用对象。
- `CollectGarbage()`：真正执行标记、弱引用处理、清扫、搬迁和引用更新。

## 5. Full GC 主体：`MarkCompactCollector::CollectGarbage`

位置：`src/mark-compact.cc:377`

主结构：

```cpp
MarkLiveObjects();
ClearWeakMaps();
SweepSpaces();
Finish();
```

完整语义：

1. 标记所有可达对象。
2. 清理 weak map 中 key 不可达的 entries。
3. 清扫未标记对象。
4. 如果启用压缩，搬迁存活对象并更新引用。
5. 收尾并切换 marking parity。

这个函数读起来很短，但它串起 full GC 的核心阶段。

## 6. Mark 阶段：`MarkLiveObjects`

位置：`src/mark-compact.cc:1919`

### 6.1 接管或终止 incremental marking

```cpp
if (was_marked_incrementally_) {
  incremental_marking->Finalize();
  incremental_marking_overflowed =
      incremental_marking->marking_deque()->overflowed();
} else {
  incremental_marking->Abort();
}
```

如果之前 incremental marking 已经做了一部分工作，full GC 不会丢掉它，而是 finalize 并接管 overflow 状态。

如果没有可用的 incremental marking，则从 stop-the-world 标记开始。

### 6.2 初始化 marking deque

```cpp
Address marking_deque_start = heap()->new_space()->FromSpacePageLow();
Address marking_deque_end = heap()->new_space()->FromSpacePageHigh();
marking_deque_.Initialize(marking_deque_start, marking_deque_end);
```

这个版本直接把 from-space 的一页拿来当 marking stack。因为 full GC 会处理整个堆，young generation 不再按普通 scavenge 方式使用 from-space。

### 6.3 从 roots 开始标记

```cpp
RootMarkingVisitor root_visitor(heap());
MarkRoots(&root_visitor);
```

`MarkRoots` 会遍历：

- strong roots；
- handle scopes；
- global handles；
- builtins；
- compilation cache；
- inactive threads；
- serialization/deserialization roots 等。

每遇到一个 heap object，就设置 mark bit，并把对象压入 marking deque。

### 6.4 扫描对象体：`EmptyMarkingDeque`

位置：`src/mark-compact.cc:1824`

```cpp
while (!marking_deque_.IsEmpty()) {
  HeapObject* object = marking_deque_.Pop();
  Map* map = object->map();
  MarkObject(map, map_mark);
  MarkCompactMarkingVisitor::IterateBody(map, object);
}
```

逻辑：

1. 从 marking deque 弹出一个黑对象。
2. 标记它的 `Map`。
3. 根据 `Map` 的 `instance_type` 扫描对象体里的指针字段。
4. 新发现的白对象会被标记并压入 deque。

这就是 mark phase 的递归闭包，只是它用显式 deque 避免 C++ 递归过深。

### 6.5 WeakMap 和弱根

标记不是简单的强引用闭包，还要处理弱引用：

```cpp
ProcessExternalMarking(&root_visitor);
global_handles()->IdentifyWeakHandles(&IsUnmarkedHeapObject);
global_handles()->IterateWeakRoots(&root_visitor);
ProcessExternalMarking(&root_visitor);
AfterMarking();
```

`JSWeakMap` 的处理在 `src/mark-compact.cc:2225`：

- 如果 WeakMap 本身可达，它会被放进 `encountered_weak_maps_` 链表。
- key 可达时，value 才被标记。
- 标记结束后，key 不可达的 entries 会被清除。

标记阶段结束后的不变量：

- 可达对象有 mark bit。
- 不可达对象没有 mark bit。
- WeakMap/weak handles 已经按语义处理。
- 后续 sweep 可以根据 mark bit 回收对象。

## 7. Sweep 阶段：`SweepSpaces`

位置：`src/mark-compact.cc:3593`

主流程：

```cpp
SweepSpace(heap()->old_pointer_space(), how_to_sweep);
SweepSpace(heap()->old_data_space(), how_to_sweep);
RemoveDeadInvalidatedCode();
SweepSpace(heap()->code_space(), PRECISE);
SweepSpace(heap()->cell_space(), PRECISE);
EvacuateNewSpaceAndCandidates();
SweepSpace(heap()->map_space(), PRECISE);
heap_->lo_space()->FreeUnmarkedObjects();
```

重点：

- old pointer/data space 可以 conservative 或 lazy sweep。
- code/cell/map space 使用 precise sweep。
- map space 最后扫，因为清扫其它对象时需要通过 map 计算对象大小；如果提前释放 dead maps，可能无法正确遍历死亡对象。
- large object space 单独释放未标记对象。

`SweepSpace()` 在 `src/mark-compact.cc:3487`。它会按 page 遍历：

- live bytes 为 0 的页可释放；
- 非 live 区间加入 free list；
- live object 清除 mark bit；
- lazy sweeping 时部分页延迟到之后 allocation 时推进。

## 8. Full GC 中的新生代和压缩：`EvacuateNewSpaceAndCandidates`

位置：`src/mark-compact.cc:2921`

这个函数处理“搬迁”和“引用修复”：

```cpp
EvacuateNewSpace();
EvacuatePages();
Update new-to-new pointers;
Update root-to-new pointers;
Update old-to-new pointers;
Update pointers to evacuated pages;
```

### 8.1 `EvacuateNewSpace`

位置：`src/mark-compact.cc:2496`

Full GC 的 young generation 不走普通 scavenge 的 roots/store-buffer 复制逻辑，因为 mark phase 已经知道哪些对象可达。

它直接遍历 from-space：

```cpp
SemiSpaceIterator from_it(from_bottom, from_top);
for (HeapObject* object = from_it.Next(); object != NULL; object = from_it.Next()) {
  MarkBit mark_bit = Marking::MarkBitFrom(object);
  if (mark_bit.Get()) {
    // live: promote or copy
  } else {
    // dead: mark map field as NULL
  }
}
```

活对象：

- 优先 `TryPromoteObject()` 晋升到 old space；
- 晋升失败则复制到另一个 semispace；
- 迁移时写 forwarding 信息。

死对象：

- 调 `LiveObjectList::ProcessNonLive(object)`；
- 把 map field 写成 `NULL`，表示它已经死亡。

### 8.2 `EvacuatePages`

位置：`src/mark-compact.cc:2628`

老生代压缩不是每次 full GC 都移动所有对象，而是先选择 evacuation candidates。候选页上的活对象会被搬迁：

```cpp
EvacuateLiveObjectsFromPage(p);
```

搬迁过程：

1. 遍历 page mark bitmap。
2. 找到 marked live object。
3. 在同一 space 中重新分配位置。
4. `MigrateObject()` 拷贝内容。
5. 旧对象写 forwarding address。

### 8.3 更新引用

搬迁后必须修复所有指针槽。`EvacuateNewSpaceAndCandidates()` 分多类更新：

```cpp
// Update pointers in to space.
SemiSpaceIterator to_it(...);
object->IterateBody(..., &updating_visitor);

// Update roots.
heap_->IterateRoots(&updating_visitor, VISIT_ALL_IN_SWEEP_NEWSPACE);

// Update old-to-new.
heap_->store_buffer()->IteratePointersToNewSpace(&UpdatePointer);

// Update slots recorded for evacuated pages.
SlotsBuffer::UpdateSlotsRecordedIn(...);
```

这里能看到几个重要 slot 来源：

- to-space 内部对象字段；
- roots；
- store buffer 中记录的 old-to-new slots；
- compaction 期间记录的 migration slots；
- code object 的 typed slots。

完成后的不变量：

- 所有外部可见引用都指向新地址。
- 被搬迁对象的旧地址只作为 forwarding 中转，之后不可再被普通代码使用。
- store buffer/slots buffer 被清理或重建到下一轮 GC 可用状态。

## 9. Write Barrier 和 StoreBuffer：为什么 Scavenge 不漏对象

位置：

- `src/objects-inl.h:923`：`WRITE_BARRIER`
- `src/objects-inl.h:932`：`CONDITIONAL_WRITE_BARRIER`
- `src/heap-inl.h:353`：`Heap::RecordWrite`

写屏障逻辑：

```cpp
heap->incremental_marking()->RecordWrite(object, slot, value);
if (heap->InNewSpace(value)) {
  heap->RecordWrite(object->address(), offset);
}
```

它有两个职责：

1. Incremental marking barrier：如果黑对象写入白对象，要维持三色不变量。
2. Generational barrier：如果 old object 指向 new object，要把 slot 记录到 store buffer。

`Heap::RecordWrite`：

```cpp
if (!InNewSpace(address)) store_buffer_.Mark(address + offset);
```

只有写入者不是 new space 时才记录。原因是 young-to-young 指针会在 young GC 扫描 to-space 对象时自然发现，不需要 remembered set。真正危险的是 old-to-new，因为 young GC 默认不扫描整个 old space。

## 10. Incremental Marking：当前版本已经有的现代化机制

位置：

- `src/incremental-marking.cc:564`：`IncrementalMarking::Start`
- `src/incremental-marking.cc:858`：`IncrementalMarking::Step`
- `src/incremental-marking-inl.h:37`：`BaseRecordWrite`

`Start()`：

```cpp
state_ = MARKING;
PatchIncrementalMarkingRecordWriteStubs(heap_, mode);
ActivateIncrementalWriteBarrier();
```

说明这个版本已经会：

- 开启 incremental marking 状态；
- patch 生成代码里的 write barrier stub；
- 给 page 设置增量标记相关 flags；
- 用 marking deque 逐步处理对象。

`Step()` 会根据 allocation rate 和 write barrier 调用次数推进标记：

```cpp
intptr_t bytes_to_process =
    marking_speed_ * Max(allocated_, write_barriers_invoked_since_last_step_);
ProcessMarkingDeque(bytes_to_process);
```

这说明 V8 3.16 的 GC 已经不是纯 stop-the-world。它已有早期 incremental marking，但还没有现代 V8 那种完整并发 marking/sweeping/work stealing 体系。

## 12. 一句话总结

当前 V8 3.16 的 GC 可以这样记：

- Young GC 是 semispace copying：从 roots/store buffer 找 new-space 活对象，复制到 to-space 或晋升到 old space，用 forwarding address 修复引用。
- Full GC 是 mark-sweep-compact：从 roots 标记全堆可达对象，处理弱引用，清扫死亡对象，搬迁新生代和压缩候选页，再统一更新所有指针槽。
- Write barrier 是连接 mutator 和 GC 的关键：它同时服务 incremental marking 和 generational remembered set。

读源码时优先沿主调用链走，不要一开始陷入所有 visitor 模板。先理解状态机、空间切换、forwarding、mark bit、store buffer、slot update，再回头看 `objects-visiting-inl.h` 里的各类对象 body descriptor。
